# ADR-007: Offline-First Architecture

**Status:** Accepted  
**Date:** 2026-07-03  
**Deciders:** Lead Software Architect

## Context

Field workers (meter readers, collectors) operate in areas with:

- No mobile coverage
- Intermittent connectivity
- Limited bandwidth

They must be able to:

- Record meter readings without internet
- Collect payments without internet
- View customer information without internet
- Sync all data when connectivity returns

Data loss is unacceptable. A single lost reading means incorrect billing, customer disputes, and revenue loss.

## Decision

We adopt an **offline-first architecture** where:

1. All data is stored locally first (SQLite via Drift).
2. The app functions fully without internet.
3. Changes are queued for sync when connectivity returns.
4. Conflicts are resolved automatically using predefined rules.

## Rationale

### Design principles

| Principle | Implementation |
|-----------|---------------|
| **Local-first storage** | Every read/write goes to SQLite first. |
| **Optimistic UI** | Changes appear immediately. Sync is background. |
| **Queue-based sync** | Outbound changes are queued with retry logic. |
| **Idempotent operations** | Sync payloads include UUIDs. Server deduplicates. |
| **Conflict resolution** | Server timestamp wins for reads. Client data preserved for payments (append-only). |

### Data flow

```
┌─────────────┐    write    ┌──────────────┐
│  UI Action   │ ─────────→ │ Local SQLite  │
└─────────────┘             └──────┬───────┘
                                   │ if online
                                   ▼
                            ┌──────────────┐
                            │  Sync Queue   │
                            └──────┬───────┘
                                   │ upload
                                   ▼
                            ┌──────────────┐
                            │  Remote API   │
                            └──────────────┘
```

### Sync strategy

| Data Type | Sync Direction | Conflict Rule |
|-----------|---------------|--------------|
| Readings | Client → Server | Client always wins (field worker was present). |
| Payments | Client → Server | Append-only. No conflicts possible. |
| Customers | Bidirectional | Last-write-wins with server timestamp. |
| Invoices | Server → Client | Server is source of truth (generated by billing engine). |
| Events | Client → Server | Append-only. No conflicts possible. |
| Settings | Server → Client | Server is source of truth. |

### Why SQLite (Drift) and not other offline solutions

| Option | Pros | Cons | Decision |
|--------|------|------|----------|
| **Drift (SQLite)** | Relational queries, migrations, type-safe, mature | Slightly more setup | ✅ Selected |
| Hive | Simple key-value, fast | No relations, no SQL queries, poor for ERP data | ❌ Rejected |
| Isar | Fast, modern | Limited query capabilities, less mature | ❌ Rejected |
| ObjectBox | Fast, reactive | Proprietary license concerns, limited community | ❌ Rejected |

ERP data is inherently relational: customers have meters, meters have readings, readings generate invoices, invoices have payments. SQLite's relational model is the natural fit.

### Sync queue structure

```
sync_queue table:
  id          UUID PRIMARY KEY
  operation   TEXT  (create, update, delete)
  entity_type TEXT  (reading, payment, customer, ...)
  entity_id   TEXT
  payload     TEXT  (JSON)
  status      TEXT  (pending, syncing, completed, failed)
  retry_count INTEGER DEFAULT 0
  created_at  DATETIME
  synced_at   DATETIME NULL
```

### Sync rules

1. Sync runs on a timer (every 5 minutes) when online.
2. Sync runs immediately when connectivity changes from offline to online.
3. Failed syncs retry with exponential backoff (max 5 retries).
4. Permanently failed syncs are flagged for manual review.
5. Sync operations are idempotent (server uses UUIDs for deduplication).
6. Large syncs are batched (max 50 records per batch).

## Consequences

- Every repository writes to local database first, regardless of connectivity.
- A `SyncManager` coordinates background sync with the server.
- A `SyncQueue` stores outbound changes with status tracking.
- UI shows sync status (last synced, pending items) in the app bar.
- Conflict resolution is automated with clear rules by entity type.
- Integration tests must cover offline scenarios.
